# Product

## Register

product

## Users

ทีม Metier (Thailand) ที่ทำงานวิเคราะห์โอกาสรับงาน + ผู้ร่วมงานฝั่งเทศบาลเมืองคลองหลวง — single workspace ไม่มี auth ทุกคนที่มี URL แก้ข้อมูลร่วมกันได้

**Context การใช้งาน:** ใช้เป็นเครื่องมือทำงานประจำวันบนเดสก์ท็อปเป็นหลัก (มี mobile บางหน้า) เปิดค้างไว้นาน ๆ ระหว่างประชุม/วางแผน หลายคนแก้พร้อมกันแบบ realtime ผู้ใช้มีหลายช่วงวัย (ที่ปรึกษา + เจ้าหน้าที่เทศบาล)

**Job to be done:** กลั่นข้อมูลแผนพัฒนาท้องถิ่น 2566–2570 (1,333 โครงการ) + ประวัติจัดซื้อจริงปี 2568 (1,449 รายการ) ให้กลายเป็นการตัดสินใจที่มั่นใจได้ว่า "Metier ควรรับโครงการไหน" แล้วติดตามโครงการที่เลือกตั้งแต่ร่าง TOR → ลำดับความสำคัญ/ไตรมาส → Kanban สถานะ → ส่งมอบ พร้อมบันทึก comment/issue ร่วมกันทั้งทีม

## Product Purpose

Webapp ภายในสำหรับเปลี่ยนข้อมูลภาครัฐปริมาณมหาศาลให้เป็น "opportunity pipeline" ที่จับต้องได้ — ตั้งแต่ค้นหา/กรอง/คัดเลือกโครงการ ไปจนถึงติดตามการส่งมอบแบบทำงานเป็นทีม realtime

**Success looks like:** ทีมเปิดแอปแล้วเห็นภาพรวมโอกาสได้ทันที ตัดสินใจรับ/ไม่รับโครงการได้อย่างมั่นใจโดยไม่ต้องเปิด Excel/PDF ดิบ และทุกคนเห็นสถานะล่าสุดตรงกันเสมอ (ไม่มีไฟล์เวอร์ชันหลงทาง)

## Brand Personality

**มั่นใจ–น่าเชื่อถือ (decision-grade).** เสียงของเครื่องมือคือผู้เชี่ยวชาญที่นิ่งและแม่นยำ — ให้ข้อมูลที่ชัดพอจะใช้ตัดสินใจรับงาน/งบประมาณได้จริง ไม่โอ้อวด ไม่เล่นมุก ไม่ตกแต่งเกินจำเป็น

3 คำ: **confident · precise · trustworthy**

Metier CI สามสีเท่านั้น — orange `#ff5008` (accent only, ห้ามใช้กับ body copy), black `#000000`, white `#ffffff` · IBM Plex Sans Thai · UI/comment ภาษาไทยเป็นหลัก

## Anti-references

ผู้ใช้ระบุชัดว่าไม่อยากให้ออกไปทาง **ทั้งสี่แบบ** ด้านล่าง — เป้าหมายคือดูเป็นเครื่องมือเฉพาะทางที่ตั้งใจออกแบบ ไม่ใช่เทมเพลตสำเร็จรูป:

- **Generic SaaS dashboard / hero-metric template** — เลขใหญ่ + gradient + การ์ดไอคอน+หัวข้อ+ข้อความเรียงเหมือนกันเป็นพรืด
- **พอร์ทัลราชการรก ๆ** — ตารางแน่นเกินอ่านไม่ออก เมนูเยอะ ลำดับความสำคัญไม่ชัด
- **แอปคอนซูเมอร์เล่น ๆ** — สีจัด การ์ตูน ลูกเล่นเยอะ ดูไม่จริงจังกับงานที่ปรึกษาภาครัฐ
- **Enterprise หนัก ๆ (SAP-like)** — เทอะทะ ดูเก่า กดยาก หลายชั้นเกิน

ห้ามใช้ (ตรงกับ impeccable absolute bans): side-stripe borders, gradient text, glassmorphism เป็น default, hero-metric template, การ์ดไอคอนเหมือนกันเรียงพรืด, eyebrow ตัวเล็ก uppercase เหนือทุก section

## Design Principles

1. **Decision-grade clarity** — ทุกหน้าต้องพาผู้ใช้ไปถึงการตัดสินใจได้ ข้อมูลสำคัญอ่านออกในวินาทีแรก ไม่กำกวม ตัวเลขงบ/สถานะ/ลำดับความสำคัญเด่นกว่าของประดับเสมอ
2. **Earn every element** — ไม่มี decoration หรือ scaffolding สำเร็จรูป ความหนาแน่นของข้อมูลต้องรับใช้ความเข้าใจ ไม่ใช่ความรก ตัดทุกอย่างที่ไม่ช่วยตัดสินใจทิ้ง
3. **Built for the work, not the demo** — นี่คือเครื่องมือใช้จริงทุกวันในเซสชันยาว ๆ เลือกความนิ่ง สงบ มืออาชีพ มากกว่าความหวือหวาที่ดูดีแค่ตอน demo
4. **Realtime truth** — เพราะทีมแก้ shared state พร้อมกันสด ๆ UI ต้องสื่อสถานะจริงตลอด (sync status, ใครแก้อะไร, สถานะปัจจุบัน) เพื่อให้ทุกคนไว้ใจข้อมูลที่เห็น
5. **Legible for everyone** — ภาษาไทย ผู้ใช้หลายช่วงวัย ความอ่านง่ายและการเข้าถึงมาก่อนความ "อิเลแกนต์" แบบตัวอักษรบาง ๆ เสมอ

## Accessibility & Inclusion

- เป้าหมาย **WCAG 2.1 AA เป็นอย่างน้อย + เน้นตัวอักษรใหญ่/อ่านง่ายเป็นพิเศษ** สำหรับผู้ใช้หลายช่วงวัย (ที่ปรึกษา + เจ้าหน้าที่เทศบาล)
- Body text ปัจจุบันตั้ง `font-weight: 300` (light) — เสี่ยงบางเกินไปสำหรับ IBM Plex Sans Thai ในขนาดเล็ก ควรทบทวนเป็น 400 และตรวจ contrast ของ `--color-muted` (#6b7280) / placeholder ให้ถึง 4.5:1
- Contrast: body ≥ 4.5:1, large text ≥ 3:1 — รวมถึง muted text บนพื้นโทนเทา/ขาวนวล
- Keyboard navigable ทุก interactive (มี orange focus ring `:focus-visible` แล้ว — รักษาไว้)
- รองรับ `prefers-reduced-motion` แล้วใน globals.css — ทุก animation ใหม่ต้องมีทางเลือก reduced motion
- ภาษาไทย: ระวัง line-height ให้พอกับสระบน/ล่าง, `text-wrap` กับหัวข้อยาว, ไม่ตัดคำผิด
